    Spfile

    SPFILE (Fichier de paramtres persistant) est un fichier binaire recherch automatiquement au dmarrage de linstance. Son nom par dfaut est spfile<SID>.ora. Il est situ dans $ORACLE_HOME/dbs/. Le fichier SPFILE est recommand par Oracle en raison de la possibilit de gr dynamiquement de nombreux paramtres dinitialisation. Il est modifi par le moteur Oracle. PFILE (Fichier de paramtres statique) est un fichier texte recherch automatiquement au dmarrage de linstance en labsence du fichier SPFILE. Il est modifi manuellement par le DBA. Son nom par dfaut est : init<SID>.ora et il est situ dans $ORACLE_HOME/dbs. Les modifications prennent effet quaprs le redmarrage de linstance. CRER UN SPFILE A PARTIR DUN PFILE Vous pouvez crer un fichier SPFILE partir dun fichier PFILE via la commande suivante (instance dmarre ou non) et ou le nom dinstance est DBA01 : 

    CREATE SPFILE = $ORACLE_HOME/dbs/spfileDBA01.ora FROM PFILE = $ORACLE_HOME/dbs/initDBA01.ora

    Control file

    Le fichier de contrle est un fichier binaire, il contient des informations sur la structure physique de la base. Il est cr pendant la cration de la base et il est modifi en permanence. Ce fichier doit tre toujours disponible car il est consult; et modifi frquemment par le serveur oracle. Et il est indispensable pour la restauration de la base. Pour afficher les noms des fichiers de contrle on utilise :

    SHOW PARAMETER CONTROL_FILES
SELECT VALUE FROM V$PARAMETER WHERE NAME='control_files';

    Redolog

    Les fichiers Redo Logs servent enregistrer toutes les modifications commites effectues sur le base de donnes. Ils servent protger la base de donnes dans le cas dun chec dinstance. En cas de restauration de la base de donnes, on utilise les redo logs pour rejouer les modifications qui ont eu lieu, les transactions valides (commit la fin de la transaction) sont enregistres dans les fichiers de donnes Oracle et les transactions non valides, sil y en a, sont effaces des fichiers de donnes.

    Data base file

    Les fichiers physiques dune base Oracle permettent de stocker de manire persistante les donnes manipules par Oracle.

    Archive log

    Quand la base est en mode NOARCHIVELOG, larchivage des fichiers redo est dsactiv. Le fichier de contrle indique que les groupes des fichiers redo pleins ne sont plus ncessaires. Ds quils sont inactifs aprs un log switch, le groupe sera disponible pour une rutilisation par la LGWR . (lgwr crit toutes les entres refaites qui ont t copies dans le tampon depuis la dernire fois.) Le mode NOARCHIVELOG protge la base contre une dfaillance dinstance et non pas contre une dfaillance media. Seulement les modifications rcentes dans la base, stocks dans les groupes de fichiers redo en ligne seront disponibles pour la restauration de la base. En mode NOARCHIVELOG, on ne peut pas sauvegarder en ligne une tablespace. Pour restaurer une base en mode NOARCHIVELOG, on est oblig de faire une sauvegarde complte quand la base est ferme. Quand la base est en mode ARCHIVELOG, larchivage des fichiers redo est activ. Le fichier de contrle de la base signale que les groupes contenant des fichiers redo pleins ne peuvent pas tre utiliss par le process LGWR tant que les groupes nont pas t archivs. Vrifier sil y a prsence dArchive log :

    Processes

    PROCESSES spcifie le nombre maximal de processus utilisateur du systme dexploitation pouvant se connecter simultanment Oracle. Sa valeur doit autoriser tous les processus darrire-plan tels que les verrous, les processus de file dattente de travaux et les processus dexcution parallles. Les valeurs par dfaut des paramtres SESSIONS et TRANSACTIONS sont drives de ce paramtre. Par consquent, si vous modifiez la valeur de PROCESSES, vous devez dterminer si vous souhaitez ajuster les valeurs de ces paramtres drivs. Afin de vrifier ltat du paramtre PROCESSES :

    SHOW parameter processes

    Application du paramtre :

    ALTER SYSTEM SET processes=[VALUE AS INTEGER] SCOPE=SPFILE ;

    Transactions

    TRANSACTIONS spcifie le nombre maximal de transactions simultanes. Des valeurs plus leves augmentent la taille du SGA et peuvent augmenter le nombre de segments de restauration allous. La valeur par dfaut est suprieure SESSIONS (et son tour, PROCESSES) pour permettre les transactions rcursives.

    Vrifier les paramtres

    Application du paramtre :

    ALTER SYSTEM SET transactions=[VALUE AS INTEGER] SCOPE=SPFILE ;

    select 'alter table '||owner||'.'||segment_name||' move;' from dba_segments where tablespace_name = 'STATSPACK_TBS' and segment_type = 'TABLE' order by bytes desc;

    How to rebuild dba_indexes :

    select 'alter index '||owner||'.'||index_name||' rebuild;' from dba_indexes where status = 'UNUSABLE';

    Tablespace

    Know the tablesapces size :

    set line 1000
set pagesize 100
col "Tablespace" FOR a22
col "Used MB" FOR 99,999,999
col "Free MB" FOR 99,999,999
col "Total MB" FOR 99,999,999

SELECT df.tablespace_name "Tablespace",
totalusedspace "Used MB",
(df.totalspace - tu.totalusedspace) "Free MB",
df.totalspace "Total MB",
round(100 * ( (df.totalspace - tu.totalusedspace)/ df.totalspace))"Pct. Free",
df.MAX_SIZE/1024/1024/1024 "Max Size Go"
FROM
(SELECT tablespace_name,
round(SUM(bytes) / 1048576) TotalSpace, 
sum(decode(AUTOEXTENSIBLE, 'YES', MAXBYTES, BYTES)) MAX_SIZE
FROM dba_data_files 
GROUP BY tablespace_name) df,
(SELECT round(SUM(bytes)/(1024*1024)) totalusedspace, tablespace_name
FROM dba_segments 
GROUP BY tablespace_name) tu
WHERE df.tablespace_name = tu.tablespace_name;
